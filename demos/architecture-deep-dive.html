Subagent Architecture & Dispatch

Sub-agent 架构的精妙之处在于工具池继承与过滤机制。父会话拥有全量工具集，每个 agent 类型通过 tools（白名单）和 disallowedTools（黑名单）声明自己需要的能力子集——Explore agent 只能读不能写，Plan agent 同理，而 general-purpose 拥有全部权限。分发时，resolveAgentTools() 从父会话的工具池中筛选出该 agent 可用的工具集，连同动态生成的 system prompt 一起传给 API。执行模式分为三种：inline（双向 SendMessage 循环，适合交互式任务）、background（注册为 LocalAgentTask 异步运行，完成后通知主会话）、remote（通过 CCR 远程执行）。Fork 机制更巧妙——它继承父会话完整的 renderedSystemPrompt 字节流来命中 prompt cache，避免重复计算，同时通过检测 <fork_boilerplate_tag> 防止递归 fork。

命令行安全机制

Bash Security & Sandbox Architecture

Bash 安全的核心挑战是：模型生成的命令本质上是不可信的用户输入，必须在执行前多层验证。第一道防线是用 tree-sitter 将命令解析为 AST（抽象语法树），如果命令过于复杂无法解析，则回退到正则匹配——这比单纯的字符串匹配更可靠，因为它能理解 shell 语法结构。第二道是 22 种注入模式检测，覆盖了 $()、反引号、进程替换等所有 shell 展开语法，以及 zsh 特有的危险内置命令。第三道是权限规则引擎，支持精确匹配、前缀通配、全通配三种模式，按 DENY > ASK > ALLOW 优先级决策。只读模式通过白名单实现——不是"禁止危险命令"，而是"只允许已知安全的命令"，并且细化到 flag 级别（比如 git checkout 允许，但带 --force 就不行）。最后一道是运行时沙盒，即使前面所有检查都通过了，命令仍然在进程级隔离环境中执行。

Defense-in-Depth: Security Layers
Layer 1 — AST-Based Security Parse
Tree-sitter parses command into AST. If too complex → fallback to regex. Each subcommand checked independently for compound commands.
Layer 2 — Command Injection Detection (22+ patterns)
Blocks: $(), ${}, backticks, <(), >(), =(), IFS injection, control chars, newline embedding, zsh dangerous builtins (zmodload, zpty, ztcp, sysopen...)
Layer 3 — Permission Rule Matching
Exact match → Prefix/wildcard → Default. Priority: DENY > ASK > ALLOW. Patterns: Bash(exact), Bash(prefix:*), Bash(*wildcard*)
Layer 4 — Read-Only Validation
Whitelist-based: 100+ safe commands. Validates flags at flag level (-x/--exec rejected). Separate lists: git, gh, docker, pyright, external tools.
Layer 5 — Sandbox Runtime
Execution-time enforcement via @anthropic-ai/sandbox-runtime. Platform-level process isolation. Final guard before shell execution.

规划与执行流程

Plan → Approve → Execute → Review

Plan 模式的设计哲学是"先想清楚再动手"，通过权限降级来强制实现。进入 Plan 模式后，toolPermissionContext.mode 被设为 'plan'，此时 Edit、Write 等写入工具被禁用，模型只能用 Glob、Grep、Read、Bash（只读）来探索代码库。计划写入磁盘文件 ~/.claude/plans/{slug}.md，slug 用词组拼接（如 "swift-noble-piano"）保证可读性。退出时有三种路径：用户自愿退出直接恢复全量权限；队友模式下通过 mailbox 发送审批请求等待 team lead 确认；UltraPlan 则更激进——在远程 CCR 会话中用 30 分钟深度规划，经历 running → needs_input → ready 三个阶段，用户可以在浏览器中编辑计划后选择本地执行（teleport 回来）或远程继续（结果直接变成 PR）。Review 同样分两档：本地用 gh CLI 快速检视，或启动 UltraReview 远程 bughunter agent 做 10-20 分钟的深度审查。

网络获取与搜索

WebFetch & WebSearch Implementation

WebFetch 和 WebSearch 虽然都是"上网"，但架构完全不同。WebFetch 是客户端实现——用 Axios 发 HTTP 请求，拿到 HTML 后用 Turndown.js 转成 Markdown，再调用 Haiku 小模型根据用户 prompt 提取关键信息，最终返回不超过 100K 字符的结果。安全上做了三层防护：先查 Anthropic 的域名黑名单 API（5 分钟缓存），再检查是否在 100+ 个预批准域名（各种文档站点）列表中，最后限制只跟随同域重定向、拒绝 URL 中的凭据。内容缓存用 LRU 策略，15 分钟 TTL、50MB 上限。WebSearch 则完全不同——它是服务端原生能力，调用 Claude API 的 web_search_20250305 beta tool，搜索在 Anthropic 服务器上执行，客户端只负责流式解析返回的 server_tool_use 和 web_search_tool_result 事件块。每个会话限 8 次搜索。两者都设置了 shouldDefer: true，意味着它们的执行可以被延迟以优化 UX。

任务管理系统

Task System & Lifecycle Management

Task 系统实际上是两套独立的机制共存。第一套是后台任务系统，管理 shell 命令、agent、工作流等异步进程的生命周期——每个任务有唯一 ID（类型前缀 + 8 字节随机数），输出流写入磁盘文件，框架每秒轮询一次读取增量，通过 circular buffer（1000 行内存）+ 8MB 溢出到磁盘的策略平衡性能和内存。卡住检测很聪明：45 秒超时后检查输出尾部是否匹配交互式提示（如 "y/n"、"Press Enter"），区分真正卡住和只是运行慢。第二套是 V2 Todo 系统，纯粹的持久化检查清单，用 JSON 文件存储在 ~/.claude/tasks/{taskListId}/，通过 proper-lockfile 实现多 agent 并发安全。两套系统在 Swarm（多 agent 团队协作）场景下汇合：任务列表以 team name 为 ID，队友共享 leader 的任务列表，通过 owner 字段追踪认领关系，writeToMailbox() 实现所有权变更通知，claimTask() 用原子忙检查防止重复认领。

上下文与记忆管理

Context Window, Compaction & Memory Architecture

上下文管理要解决的核心矛盾是：对话可以无限长，但模型的上下文窗口是有限的（200K 或 1M tokens）。系统用三层策略应对。第一层是精确的 token 预算追踪——tokenCountWithEstimation() 实时计算当前上下文占用，当距离窗口上限还剩 13K tokens 时触发自动压缩。第二层是对话压缩（Compaction）——启动一个 forked sub-agent 将历史对话摘要为不超过 20K tokens 的总结，插入 SystemCompactBoundaryMessage 标记，旧消息被丢弃但关键信息保留。第三层是持久化记忆——memdir/ 系统维护一个 MEMORY.md 索引文件（限 200 行 / 25KB）指向独立的记忆文件，每次对话开始时用 Sonnet 模型语义匹配最多 5 条相关记忆注入上下文。还有 SessionMemory 在后台周期性运行，当上下文用量超过 50% 时开始记录会话笔记到 .claude-sessions/{sessionId}/session-memory.md，确保即使发生压缩也不会丢失关键上下文。

团队模式与协同

Coordinator / Teammate / Swarm Architecture

团队模式的本质是让多个 AI agent 像人类团队一样协作，有明确的角色分工和通信机制。系统支持两种模式：Coordinator 模式是上下级关系——Coordinator 作为 supervisor 通过 Agent tool 派发 worker，worker 只有有限的工具访问权（Bash、Read、Edit），不能再创建团队或发消息，完成后通过 task notification 汇报；Team 模式是对等的 swarm——通过 TeamCreate 创建团队，成员信息存储在 ~/.claude/teams/{team_name}/，每个 teammate 有独立的 inbox JSON 文件实现异步通信。通信用文件锁（proper-lockfile）保证并发安全，支持 10+ agent 同时操作。Teammate 可以在同一进程内运行（AsyncLocalStorage 隔离上下文）或在外部终端面板运行（tmux/iTerm2）。权限流最巧妙：teammate 要执行危险操作时，将 permission_request 写入 leader 的 mailbox，leader 在自己的 UI 上弹出对话框让用户决策，结果写回 teammate 的 mailbox——整个链路中用户始终保有最终控制权。

Complete Internal Command Reference

Claude Code 的命令系统分为三类：Standard 是所有用户可用的命令，Feature-Gated 需要特定功能开关或订阅才能使用，Internal 仅限 Anthropic 内部员工（通过 USER_TYPE = 'ant' 判断）。命令本身也有三种类型：local-jsx 渲染交互式 Ink UI 组件（如 /config 打开设置面板），local 输出纯文本（如 /cost 显示费用），prompt 展开为发送给模型的 prompt（本质上就是 Skill）。每个命令可以声明 aliases（别名）、availability（需要的认证类型）、isHidden（隐藏不显示在帮助中）、isImmediate（跳过确认直接执行）等属性。
